# v1.3.23 Mission Objective Registry + Status Contract

Status: active/implemented for Core Stabilization Round A (2026-02-15)

## Purpose

Define a canonical mission-objective manifest and a read-only status surface that
release gates can consume without parsing test logs ad hoc.

## Registry Artifact

- Canonical manifest path:
  - `core/config/v1_3_23_mission_objectives.json`
- Owner:
  - `core` (system-of-record for objective identity and pass/fail semantics)

## Objective Schema (v1.3.23)

Each objective record must include:

- `id` (stable key; e.g. `obj.v1_3_23.replay_determinism`)
- `description` (human-readable gate requirement)
- `owner` (`core`, `wizard`, `ci`, or `cross-surface`)
- `severity` (`blocker`, `required`, or `advisory`)
- `checks` (list of machine checks; pytest path and/or CI script id)
- `threshold` (optional numeric or structured threshold contract)
- `status` (`pending`, `pass`, `fail`, `error`)
- `updated_at` (UTC ISO-8601)
- `evidence` (list of short evidence records; command + summary)

## Status Endpoint Contract

Expose a machine-readable status view for release gates:

- Command surface:
  - `HEALTH CHECK release-gates --format json`
- Payload root:
  - `version`, `generated_at`, `milestone`, `summary`, `objectives`
- `summary` fields:
  - `total`, `pass`, `fail`, `error`, `pending`, `blocker_open`

## Determinism Rules

- Objective `id` values are immutable once published.
- Unknown objective ids in status payload are treated as contract drift.
- Missing required objectives are hard failures for v1.3.23 release gate.
